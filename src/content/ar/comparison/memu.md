# OpenLegion مقابل MemU: إطار وكلاء كامل مقابل طبقة ذاكرة متخصصة

MemU ليس إطار وكلاء منافساً — إنه نظام ذاكرة دائم متخصص لوكلاء الذكاء الاصطناعي. فهم هذا التمييز ضروري: يوفّر MemU "الدماغ" (ذاكرة منظمة تتطوّر مع الوقت)، بينما توفّر أطر مثل OpenLegion "الجسد" (بيئة التنفيذ، التنسيق، الأمان، وصول الأدوات). يحلّان مشاكل مختلفة، وفي كثير من الحالات، يمكن أن يكونا متكاملين.

أنشأت NevaMind AI MemU ونما إلى نحو 7,200-10,500 نجمة GitHub. يعامل الذاكرة كنظام ملفات هرمي مع تنظيم ذكي، وربط متبادل، وتطوّر، وتقليم. المنتج المرافق memUBot (167 نجمة) يضع نفسه كـ "OpenClaw جاهز للمؤسسات" يجمع ذاكرة MemU مع بيئة تشغيل وكلاء.

OpenLegion هو [إطار وكلاء ذكاء اصطناعي](/learn/ai-agent-platform) يضع الأمان أولاً مع عزل حاوية Docker إلزامي، وإدارة بيانات اعتماد عبر وكيل الخزنة، وفرض ميزانية لكل وكيل، وتنسيق نموذج الأسطول (سبورة + نشر/اشتراك + تسليم)، وذاكرة دائمة مدمجة لكل وكيل.

> **ما الفرق بين OpenLegion وMemU؟**
> MemU هو إطار ذاكرة وكيلي متخصص يوفّر ذاكرة دائمة ومنظمة ومتطوّرة لوكلاء الذكاء الاصطناعي — يقع بين LLM وطبقة التطبيق كمكوّن ذاكرة قابل للإضافة. OpenLegion هو إطار وكلاء كامل مع تنفيذ، وتنسيق، وأمان، وذاكرة دائمة مدمجة لكل وكيل. يوفّر MemU الذاكرة للوكلاء المبنية على أطر أخرى؛ يشمل OpenLegion الذاكرة كجزء من منصة متكاملة تضع الأمان أولاً.

## اختر MemU إذا...

**تحتاج إلى نظام ذاكرة مخصّص ومتطوّر.** نموذج ذاكرة MemU أكثر تقدّماً من أي ذاكرة مدمجة في إطار. استعارة نظام الملفات الهرمي (الفئات كمجلدات، العناصر كملفات، الروابط المتبادلة كـ symlinks)، مدمجة مع أربع آليات أساسية — Organize وLink (رسم بياني للمعرفة) وEvolve (تأمل ذاتي غير متصل) وForget (تقليم ذكي) — توفّر قدرات ذاكرة لا يضاهيها أي إطار وكلاء أصلياً.

**وكلاؤك يعملون على إطار مختلف.** صُمّم MemU كمكوّن قابل للإضافة. إذا كنت تبني على LangGraph أو CrewAI أو AutoGen أو أي إطار آخر وتحتاج إلى ذاكرة دائمة تستمر بعد الجلسات الفردية، يتكامل MemU عبر Python SDK أو REST API.

**جودة الذاكرة تهم أكثر من بساطة الذاكرة.** الاسترجاع ثنائي الوضع في MemU — Fast Context (تشابه متجهي رخيص للمراقبة) وDeep Reasoning (استدلال LLM كامل يُحفَّز فقط عند اكتشاف الصلة) — نهج ذكي يوازن التكلفة بالجودة. يدّعي دقة 92.09% على معيار Locomo.

**تحتاج إلى ذاكرة تتطوّر بشكل مستقل.** تشغّل آلية Evolve في MemU تأملاً ذاتياً غير متصل على الذكريات المخزّنة، وتنشئ رؤى جديدة وروابط متبادلة دون توجيه المستخدم. هذه قدرة غير متاحة في أي ذاكرة مدمجة في إطار.

## اختر OpenLegion إذا...

**تحتاج إلى إطار وكلاء كامل، وليس مكوّن ذاكرة.** MemU لا يبني، أو ينشر، أو يعزل، أو ينسّق الوكلاء. يوفّر الذاكرة للوكلاء المبنية على أطر أخرى. OpenLegion هي منصة كاملة: تنفيذ الوكلاء، وعزل حاوية Docker، وبيانات اعتماد عبر وكيل الخزنة، وفرض الميزانية، وتنسيق نموذج الأسطول، وإدارة الأدوات، وذاكرة دائمة مدمجة.

**بساطة بنية الذاكرة التحتية تهم.** يستخدم OpenLegion ذاكرة SQLite مضمّنة — لا قاعدة بيانات خارجية مطلوبة. يتطلب MemU PostgreSQL مع امتداد pgvector، مما يضيف تعقيداً تشغيلياً (توفير قاعدة البيانات، النسخ الاحتياطية، إدارة الاتصالات، التحجيم).

**تحتاج إلى ذاكرة مع عزل أمان لكل وكيل.** ذاكرة OpenLegion لكل وكيل ومعزولة بحدود الحاوية. لا يمكن للوكيل A الوصول إلى ذاكرة الوكيل B. ذاكرة MemU يمكن الوصول إليها عبر API — يعتمد التحكم في الوصول على تنفيذ إطار المضيف.

**تحتاج إلى تحكم متكامل في التكلفة عبر الذاكرة والتنفيذ.** تشمل ميزانية كل وكيل في OpenLegion جميع التكاليف (استدعاءات LLM، استخدام الأدوات، عمليات الذاكرة). يفوتر MemU بشكل منفصل عن إطار المضيف — تستهلك استدعاءات الذاكرة مجموعة رصيدها الخاصة، مما يجعل تتبع التكلفة الإجمالية أكثر تعقيداً.

**تريد بائعاً واحداً للبنية التحتية للوكلاء.** يوفّر OpenLegion إطار + ذاكرة + أمان + تنسيق في حزمة واحدة. يتطلب MemU دمجه مع إطار وكلاء منفصل، وطبقة أمان، ونظام تنسيق.

## مقارنة بنية الذاكرة

### الذاكرة المدمجة في OpenLegion

يوفّر OpenLegion ذاكرة دائمة لكل وكيل باستخدام SQLite مضمّن مع بحث متجهي. لكل وكيل في سير عمل تنسيق نموذج الأسطول تخزين ذاكرة معزول يستمر عبر التنفيذات. الذاكرة محصورة بالوكيل — ذكريات الوكيل A غير مرئية للوكيل B ما لم تُشارَك صراحةً عبر مخرجات سير العمل. نظام الذاكرة يعمل لحالات استخدام الوكلاء النموذجية (تاريخ المحادثة، سياق المهمة، التفضيلات المتعلّمة) دون اعتماديات خارجية.

### الذاكرة المتخصصة في MemU

يعامل MemU الذاكرة كبنية بيانات من الدرجة الأولى بأربع آليات:

**Organize** يصنّف المعلومات الواردة إلى بنية هرمية تلقائياً. تُحفظ الذكريات الجديدة في الفئات المناسبة دون وسم يدوي.

**Link** ينشئ رسماً بيانياً للمعرفة من المراجع المتبادلة بين الذكريات. عندما ترتبط ذاكرة جديدة بذكريات موجودة، ينشئ MemU روابط ثنائية الاتجاه — يبني شبكة من الترابطات تحسّن دقة الاسترجاع.

**Evolve** يشغّل تأملاً ذاتياً غير متصل. دون توجيه المستخدم، يعيد MemU دورياً فحص الذكريات المخزّنة، ويولّد رؤى جديدة، ويحدد الأنماط، وينشئ ذكريات اصطناعية تلتقط فهماً من ترتيب أعلى.

**Forget** ينفّذ تقليماً ذكياً. بدلاً من الاحتفاظ بكل شيء إلى الأبد، يحدد MemU الذكريات الزائدة عن الحاجة، أو القديمة، أو منخفضة الصلة، ويقلّمها — مع إبقاء نظام الذاكرة مركّزاً وفعّالاً من حيث التكلفة.

يحسّن الاسترجاع ثنائي الوضع (Fast Context للمراقبة، Deep Reasoning عند اكتشاف الصلة) مقايضة التكلفة-الجودة. دقة معيار Locomo البالغة 92.09% أعلى بشكل كبير من تطبيقات RAG النموذجية.

### المقايضة

ذاكرة MemU موضوعياً أكثر تطوراً. ذاكرة OpenLegion أبسط، ومتكاملة، ومعزولة لكل وكيل دون اعتماديات خارجية. للفرق التي تحتاج إلى قدرات ذاكرة متقدمة، يمكن دمج MemU مع OpenLegion كخلفية ذاكرة — استبدال ذاكرة SQLite المدمجة بـ API الخاصة بـ MemU.

## نظام MemU البيئي: ما يفعله بشكل أفضل

### منتج memUBot الكامل

تطوّر NevaMind AI أيضاً memUBot (github.com/NevaMind-AI/memUBot، 167 نجمة)، الذي يضع نفسه كـ "OpenClaw الجاهز للمؤسسات" — مساعد ذكاء اصطناعي استباقي يجمع ذاكرة MemU مع بيئة تشغيل وكلاء. memUBot هو المنتج الكامل؛ MemU هي طبقة الذاكرة غير المجمَّعة.

### أنماط التكامل

يتكامل MemU مع أي تطبيق Python عبر `pip install memu-py` أو أي لغة عبر REST API. تشمل الأنماط الشائعة: إضافة ذاكرة دائمة لوكلاء LangChain، ومنح فرق CrewAI استدعاءً طويل الأمد، وزيادة وكلاء OpenClaw/NanoClaw بذاكرة منظمة، وبناء وكلاء مخصّصين يحتاجون إلى التذكّر عبر الجلسات.

### API السحاب (memu.pro)

يقدّم MemU API مستضافاً على memu.pro بتسعير قائم على الاستخدام: مجاناً (30 استدعاء ذاكرة)، Professional (600 استدعاء)، Enterprise (SSO/RBAC). إصدار مجتمع مستضاف ذاتياً "قريباً". يوفّر نموذج SaaS هذا الراحة لكنه يعني أن بيانات الذاكرة تعبر خدمة خارجية.

### مخاوف الإنتاج الشائعة

**ترخيص AGPL-3.0.** ترخيص الخادم هو AGPL-3.0، الذي يتطلب توزيع شيفرة المصدر لأي إصدارات معدلة وأي برامج تتفاعل مع MemU عبر شبكة (حسب التفسير). تتجنب العديد من المؤسسات AGPL. هذا ترخيص أكثر تقييداً بشكل كبير من PolyForm Perimeter License 1.0.1 في OpenLegion أو تراخيص MIT/Apache في معظم المنافسين.

**اعتمادية قاعدة بيانات خارجية.** يضيف PostgreSQL + pgvector تعقيداً تشغيلياً. توفير قاعدة البيانات، وتجميع الاتصالات، والنسخ الاحتياطية، والتحجيم مسؤوليات إضافية.

**إقامة بيانات الذاكرة.** إذا كنت تستخدم API السحاب، تُخزَّن بيانات الذاكرة (التي قد تحتوي على معلومات مستخدم حساسة، وتاريخ محادثة، وأنماط متعلّمة) على بنية MemU التحتية. للصناعات الخاضعة للتنظيم، قد يكون هذا مشكلة امتثال.

**تعقيد نموذج التكلفة.** يفوتر MemU لكل استدعاء ذاكرة، بينما يفوتر إطار المضيف بشكل منفصل لاستدعاءات LLM، واستخدام الأدوات، والتنفيذ. يتطلب تتبع التكلفة الإجمالية ربط نظامي فوترة.

### ما يغطيه OpenLegion بشكل مختلف

يدمج OpenLegion الذاكرة في نموذج أمانه: عزل ذاكرة لكل وكيل (يفرضه حدود الحاوية)، وذاكرة مدرجة في محاسبة ميزانية كل وكيل، ولا اعتمادية قاعدة بيانات خارجية، ولا بيانات تغادر بيئة النشر. الذاكرة أبسط لكنها مؤمَّنة بنفس البنية التي تحمي بيانات الاعتماد وتفرض حدود التكلفة.

## مقايضات الاستضافة مقابل الاستضافة الذاتية

**MemU** يقدّم API سحاب (memu.pro) أو نشراً مستضافاً ذاتياً يتطلب PostgreSQL مع pgvector. API السحاب هو أسرع مسار لكنه يرسل بيانات الذاكرة إلى بنية تحتية خارجية. تتطلب الاستضافة الذاتية إدارة قاعدة البيانات.

**OpenLegion** يشمل الذاكرة كـ SQLite مضمّن — لا خدمات خارجية، ولا إدارة قاعدة بيانات، ولا بيانات تغادر النشر. تشمل المنصة المستضافة بنية تحتية للذاكرة.

## لمن هو

**MemU** للمطوّرين الذين يبنون على أطر وكلاء موجودة ويحتاجون إلى ذاكرة دائمة ومتطوّرة تتجاوز ما يوفّره إطارهم. المستخدم المثالي لديه وكلاء على LangChain أو CrewAI أو إطار مخصّص ويريد إضافة ذاكرة طويلة الأمد منظمة دون بنائها من الصفر. ذو قيمة أيضاً للباحثين الذين يدرسون بنى ذاكرة الوكلاء.

**OpenLegion** للفرق التي تحتاج إلى إطار وكلاء كامل مع أمان وتنسيق وذاكرة متكاملة. المستخدم المثالي يريد نظاماً واحداً يتعامل مع التنفيذ، وبيانات الاعتماد، والميزانيات، وسير العمل، والذاكرة — دون تجميع مكوّنات من بائعين متعددين.

## المقايضة الصادقة

ذاكرة MemU أكثر تطوراً من ذاكرة OpenLegion المدمجة. خط أنابيب Organize-Link-Evolve-Forget، والاسترجاع ثنائي الوضع، ودقة Locomo 92% تمثّل ابتكاراً حقيقياً في ذاكرة الوكلاء.

لكن MemU مكوّن، وليس منصة. لا يحلّ إدارة بيانات الاعتماد، أو عزل الوكلاء، أو التحكم في التكلفة، أو تنسيق سير العمل. ذاكرة OpenLegion أبسط لكنها موجودة داخل إطار أمان يحميها — معزولة لكل وكيل، مدرجة في محاسبة الميزانية، ولا تتطلب اعتماديات خارجية.

للفرق التي تحتاج إلى ذاكرة متقدمة على إطار موجود، استخدم MemU. للفرق التي تحتاج إلى إطار وكلاء كامل وآمن بذاكرة مدمجة كافية، استخدم OpenLegion. للفرق التي تريد كليهما، يمكن دمج MemU كخلفية ذاكرة لـ OpenLegion.

للمشهد الكامل، راجع [مقارنة أطر وكلاء الذكاء الاصطناعي](/learn/ai-agent-frameworks).

## الأسئلة المتكررة

### ما هو MemU؟

MemU هو إطار ذاكرة وكيلي متخصص أنشأته NevaMind AI. يوفّر ذاكرة دائمة ومنظمة ومتطوّرة لوكلاء الذكاء الاصطناعي باستخدام استعارة نظام ملفات هرمي بأربع آليات: Organize، Link، Evolve، وForget. يدّعي دقة 92.09% على معيار Locomo ومتاح عبر Python SDK أو REST API أو خدمة سحاب (memu.pro). لديه نحو 7,200-10,500 نجمة GitHub.

### OpenLegion مقابل MemU: ما الفرق؟

MemU هو طبقة ذاكرة متخصصة — يوفّر ذاكرة دائمة للوكلاء المبنية على أطر أخرى. OpenLegion هو إطار وكلاء كامل مع تنفيذ، وأمان، وتنسيق، وذاكرة مدمجة. يحلّان مشاكل مختلفة. يوفّر MemU ذاكرة أكثر تطوراً؛ يوفّر OpenLegion ذاكرة متكاملة داخل منصة تضع الأمان أولاً.

### هل OpenLegion بديل لـ MemU؟

يشمل OpenLegion ذاكرة دائمة مدمجة لكل وكيل، لذا يمكن أن يخدم كبديل لـ MemU للفرق التي تحتاج إلى ذاكرة كافية (وليست متقدمة) داخل إطار وكلاء كامل. للفرق التي تحتاج تحديداً إلى قدرات Evolve وLink المتقدمة في MemU، يظل MemU نظام الذاكرة الأكثر قدرة — يمكن استخدامه بشكل محتمل إلى جانب OpenLegion.

### كيف تقارَن معالجة الذاكرة بين OpenLegion وMemU؟

يستخدم OpenLegion SQLite لكل وكيل مع بحث متجهي — بسيط، مضمّن، معزول لكل حاوية، لا اعتماديات خارجية. يستخدم MemU PostgreSQL + pgvector مع تنظيم هرمي، وربط رسم بياني للمعرفة، وتطوّر مستقل، وتقليم ذكي. MemU أكثر تطوراً؛ OpenLegion أبسط وأكثر أماناً (الذاكرة معزولة بحدود الحاوية، لا خروج بيانات خارجي).

### أيهما أفضل لوكلاء الذكاء الاصطناعي للإنتاج؟

يخدمان احتياجات مختلفة. MemU أفضل لمتطلبات ذاكرة الإنتاج (استرجاع معقّد، معرفة متطوّرة، ربط متبادل). OpenLegion أفضل لمتطلبات أمان الإنتاج (عزل بيانات الاعتماد، عزل الحاويات، فرض الميزانية، تنسيق نموذج أسطول قابل للتدقيق). قد يستخدم مكدّس الإنتاج المثالي كليهما.

### هل يوفّر MemU عزل الوكلاء أو الأمان؟

لا. MemU هو طبقة ذاكرة — لا يبني، أو ينشر، أو يعزل، أو ينسّق الوكلاء. الأمان (إدارة بيانات الاعتماد، عزل التنفيذ، التحكم في الوصول) هو مسؤولية إطار المضيف. يوفّر OpenLegion طبقات الأمان هذه بشكل أصلي.

### هل يمكن استخدام MemU مع OpenLegion؟

بشكل محتمل. يمكن أن يخدم API الخاص بـ MemU كخلفية ذاكرة خارجية لوكلاء OpenLegion. يجمع هذا بين ذاكرة MemU المتقدمة وبنية OpenLegion التحتية للأمان. هذا التكامل ليس مدمجاً حالياً لكنه ممكن معمارياً.
